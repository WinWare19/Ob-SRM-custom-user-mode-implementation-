# Ob-SRM-custom-user-mode-implementation-
a user mode implementation of two major kernel executive managers, the purpose if this repository is to help anybody understand how objects are managed internally in windows and how the security layer that protect obejcts from unathorized access works. The implmentation is in user mode so you can easily test it.
# Object Manager
a kernel executive manager responsible for the creation and deletion of objects and lifecycles tracking through reference counting, it has a private namespace used to store object names ( some objects are unnamed ), each object belongs to a specific type which defines the valid access rights and many other common pieces of information. Objects has headers where metadata are stored such as current reference and handle counts, security descriptor .. etc.
Objects are considered protected resources so user mode clients cannot access them directly, although they must use an indirect reference called a HANDLE, each handle points to exactly a single object, the handle is simply a value that's used as an index in the current process' handle table, each handle entry stores the granted access rights and a pointer to the start of the obejct and many other pieces of informtion.
# Security Reference Monitor 
a kernel executive manager responsible for ensuring secure access to securable objects ( objects that have a security descriptor ), it supports many types of objects like access tokens which are used to provide the security identity of the current task context, security identities are repressented using a SID, each account in the system has its own unique SID, access tokens also contains granted privileges to the account the token is representing. Objects are secured using a security descriptor which a data structre that provide the owner of the object, its primary group ( generally got from the creator access token ), a DACL which contains granted and denied access rights to the object and finaly a SACL that determines which access rights should be audited and under what condition ( success, failure or both ). Before the access check process starts the security reference monitor checks if the access token.s integrity level is equal o higher than the obejct's one ( found in its security descriptor ),
